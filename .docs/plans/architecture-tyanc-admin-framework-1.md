# Introduction

![Status: In progress](https://img.shields.io/badge/status-In%20progress-yellow)

This revision updates the Tyanc administration-framework plan to align with the clarified product goal: Tyanc is an installable admin foundation for future real-world apps, and the Tyanc app is the single governance plane for users, roles, permissions, app registration, and internal route or module access management. This revision also removes CMS or public page-management scope from Tyanc core; public-facing content management belongs to future business apps that plug into Tyanc. Phases 1 through 5 are treated as the locked implementation baseline and are intentionally left unchanged in this revision. New planning work starts after that baseline, adding app-aware RBAC, a first-class app registry, and dedicated Tyanc governance pages for apps, roles, permissions, and access-matrix management. Any cross-cutting behavior that touches the already-implemented shell, user-management, middleware, audit, or seed foundations from phases 1 through 5 must be implemented as an extension in phases 6 and later instead of reopening those baseline phases. The current codebase now also isolates Tyanc-only backend actions and DTOs under `app/Actions/Tyanc/**` and `app/Data/Tyanc/**`, keeps shared platform code in shared locations, groups Tyanc admin routes in `routes/tyanc.php`, and groups Tyanc frontend types in `resources/js/types/tyanc/**`; all future plan steps must follow that structure.

## 1. Requirements & Constraints

- **REQ-001**: Preserve the current Laravel 13, PHP 8.5, Inertia.js v3, Vue 3, Tailwind CSS v4, and Wayfinder stack already present in the repository.
- **REQ-002**: Keep public pages, authentication routes, and general authenticated user settings routes on the root domain inside `routes/web.php`.
- **REQ-003**: Preserve the current authenticated multi-app topology where the Tyanc application lives under `/tyanc/*`, the sandbox lives under `/demo/*`, and `/dashboard` redirects to `/tyanc/dashboard`.
- **REQ-004**: Preserve the sidebar app switcher between `tyanc` and `demo`, with the selected app exposed to the frontend as `page.props.currentApp`.
- **REQ-005**: Preserve `page.props.sidebarNavigation` as the only frontend navigation source for app-specific sidebar menus.
- **REQ-006**: Keep app selection persisted in the `current_app` cookie and normalized in `App\Http\Middleware\HandleInertiaRequests::share()` based on route names.
- **REQ-007**: Keep navigation definitions centralized in `config/sidebar-menu.php` and resolved by `App\Actions\ResolveSidebarNavigation::handle(string $currentApp): array`.
- **REQ-008**: Scaffold headless API endpoints under the `api.tyanc.test/api/v1` boundary by registering `routes/api.php` and versioned API controllers.
- **REQ-009**: Implement DTO-first request and response handling by introducing Spatie Laravel Data classes for Inertia props and future JSON responses.
- **REQ-010**: Enforce strict schema-to-form parity for every field added to `users`, `user_profiles`, settings payloads, files, approvals, conversations, and messages.
- **REQ-011**: Use shadcn-vue components exclusively for interactive UI primitives; use Dialogs for CRUD forms, Sheets for settings and filters, and Sonner or equivalent toaster notifications for feedback.
- **REQ-012**: Replace placeholder dashboard and table experiences with reusable TanStack Table based DataTable components that synchronize filters, sorting, pagination, and column visibility with the Inertia URL state.
- **REQ-013**: Implement RBAC with Spatie Laravel Permission, reserved `Supa Manuse` and `Manuse` roles, and a custom role hierarchy using a persisted `level` integer.
- **REQ-014**: Extend the user domain to include authentication state, avatar, status, locale, timezone, last login telemetry, soft deletes, and a one-to-one `user_profiles` record.
- **REQ-015**: Implement brand, appearance, security, and global user-default settings with Spatie Laravel Settings; persist per-user preference overrides in a dedicated `user_preferences` table; inject resolved CSS variables through shared middleware data.
- **REQ-016**: Build all authentication pages in Vue/Inertia, but leave only registration and login enabled by default in `config/fortify.php`.
- **REQ-017**: Support media uploads, import/export, PDF generation, audit logging, notifications, approvals, files, and internal messaging with the packages named in the PRD.
- **REQ-018**: Seed development and demo data with humanized Faker values only; remove starter-kit placeholder data and random gibberish generators.
- **REQ-019**: Implement Laravel-native localization with JSON translation files at `lang/en.json` and `lang/id.json`; set `config/app.php` so `locale` and `fallback_locale` are both `en`.
- **REQ-020**: Resolve the active locale on every browser request through `App\Http\Middleware\SetLocale::handle(Request $request, Closure $next): Response` by checking the authenticated user preference first, the session locale second, and the global application default last.
- **REQ-021**: Share `locale`, `availableLocales`, and contextual `translations` through `App\Http\Middleware\HandleInertiaRequests::share(Request $request): array`; no Tyanc or Demo frontend screen may hardcode user-facing static copy.
- **REQ-022**: Expose the default system language in Tyanc application settings and expose the per-user locale override in personal preferences and profile-management flows.
- **REQ-023**: Use localized Faker generators for locale-sensitive seed data, including `Faker\Factory::create('id_ID')` for Indonesian demo and workflow fixtures.
- **REQ-024**: Treat Tyanc as an installable foundation platform for future applications, not a one-off admin panel; the Tyanc app under `/tyanc/*` is the single governance plane for all registered apps.
- **REQ-025**: Introduce a first-class app registry with at least `key`, `label`, `route_prefix`, `icon`, `permission_namespace`, `enabled`, `sort_order`, and `is_system` metadata so future apps such as `erp` and `tasks` can be added without restructuring the shell.
- **REQ-026**: Use Spatie Laravel Permission as the installed RBAC engine; use app-scoped permission names in the format `<app>.<resource>.<action>`; define allowed apps, resources, and actions in `config/permission-sot.php`; treat roles as the primary source of permissions; allow users to have multiple roles; and treat direct user permissions as an exception-only mechanism.
- **REQ-027**: Centralize governance UI inside Tyanc at `/tyanc/apps`, `/tyanc/roles`, `/tyanc/permissions`, and `/tyanc/access-matrix`; keep `/tyanc/permissions` as a read-only generated catalog and sync-status screen, not a manual permission CRUD surface.
- **REQ-028**: Role creation and permission assignment must be separate workflows; administrators create or edit role metadata first, then assign permissions from a dedicated flow that filters by app and resource and persists the derived full permission names through Spatie.
- **REQ-029**: Enforce access at the app, route or module, and action layers; unauthorized apps and internal navigation targets must be hidden from navigation and still return authorization failures on direct access.
- **REQ-030**: Deliver file-management, import, export, PDF, and approval workflows for Tyanc itself first, while keeping the underlying actions reusable for future apps.
- **SEC-001**: Apply a global `Gate::before` bypass for `Supa Manuse` users and require explicit permissions for every `Manuse` action.
- **SEC-002**: Keep sensitive fields hidden, validated, and encrypted where applicable, including passwords, TOTP secrets, recovery codes, uploaded assets, and session/security settings.
- **SEC-003**: Shared routes such as `/settings/*` must resolve sidebar context from `current_app` unless the current route explicitly belongs to the `demo.*` namespace or the Tyanc dashboard route.
- **SEC-004**: Accept only `en` and `id` as writable locale values for profile updates, user preferences, application settings, and session-based language switching.
- **SEC-005**: Prevent non-super administrators from creating, updating, deleting, or assigning roles above their hierarchy level, and prevent them from granting permissions outside their allowed scope or outside the combinations declared in `config/permission-sot.php`.
- **SEC-006**: Do not allow disabling or deleting reserved system apps or roles such as `tyanc`, `demo`, `Supa Manuse`, and `Manuse` unless a future migration explicitly changes that rule.
- **SEC-007**: Treat navigation visibility and route protection as two separate enforcement layers; a hidden page is not considered secure unless server-side authorization still denies direct access.
- **CON-001**: Do not replace the monolith architecture or the current app-aware shell with a separate SPA API or a third-party admin framework.
- **CON-002**: Follow the Action pattern already used in `app/Actions`, with business logic in dedicated action classes exposing a single `handle()` method.
- **CON-003**: Follow existing repository conventions for Vue page placement under `resources/js/pages`, layouts under `resources/js/layouts`, and reusable UI primitives under `resources/js/components`.
- **CON-004**: Keep every implementation step compatible with Wayfinder route generation and typed frontend imports from `@/routes`, `@/routes/demo`, and future generated actions.
- **CON-005**: Treat the work as an XL multi-phase delivery; only start a phase after every dependency noted in its tasks is complete.
- **CON-006**: Do not introduce a second translation source such as Vue I18n or manually maintained frontend locale JSON files; Laravel JSON translations remain the only source of truth.
- **CON-007**: Phases 1 through 5 are locked baselines in this revision and must remain unchanged while the missing Tyanc governance and app-aware RBAC work is planned after them.
- **CON-008**: Preserve the current shared Inertia prop contract from `App\Http\Middleware\HandleInertiaRequests::share()`, including `currentApp`, `sidebarNavigation`, `sidebarOpen`, `brand`, `theme`, `locale`, `availableLocales`, `translations`, `userPreferences`, `notifications`, and `auth.user`; later phases may extend this payload but must not rename or remove the existing keys.
- **GUD-001**: Use grayscale-first UI tokens, pastel accents, modern sans-serif typography, and same-color opacity gradients only.
- **GUD-002**: Remove all purple usage from `resources/css/app.css`, chart palettes, badges, gradients, and page accents.
- **GUD-003**: Maintain zero dead code tolerance by deleting starter placeholders, unused imports, orphaned pages, and superseded helpers once replacement features are live.
- **GUD-004**: Prefer translated labels, button text, empty states, helper text, and notifications across every Tyanc and Demo page so both supported locales remain complete and visually consistent.
- **PAT-001**: Negotiate web and JSON responses in controllers with `request()->wantsJson()` and Data DTOs instead of passing raw Eloquent models directly to Inertia or API responses.
- **PAT-002**: Model complex write workflows through dedicated actions and `DB::transaction()` when multiple models or media records mutate together.
- **PAT-003**: Keep sidebar navigation app-aware by adding all new app entries to `config/sidebar-menu.php`, resolving them in `App\Actions\ResolveSidebarNavigation`, mapping them in `resources/js/lib/sidebar-navigation.ts`, and consuming them through `resources/js/composables/useAppNavigation.ts`.
- **PAT-004**: Use generated route helpers instead of hardcoded frontend URLs.
- **PAT-005**: Load only route-appropriate translation slices through a dedicated resolver such as `App\Actions\ResolveTranslations::handle(string $routeName, string $locale): array` and expose the result through shared Inertia props.
- **PAT-006**: Register a global Vue translation helper in `resources/js/app.ts` from `resources/js/lib/translations.ts` so components can call `__('Some text')` against `usePage().props.translations`.
- **PAT-007**: Resolve accessible apps, internal routes or modules, and navigation items from a single app-registry plus permission-matrix flow so the app switcher, sidebar, policies, and route middleware stay aligned.
- **PAT-008**: Prefer checking permission names resolved through a shared config-backed permission-name helper in policies, middleware, controllers, and views instead of checking role names directly, except for reserved-role bypass and reserved-role protections.
- **PAT-009**: Keep direct user permissions as a last-resort override; the normal workflow is role assignment, role-permission assignment, and permission-based authorization checks.
- **PAT-010**: Represent internal route and module access through a shared access-matrix domain or resolver rather than duplicating scattered hardcoded checks per screen.
- **PAT-011**: Migrate the existing legacy Tyanc permission names `manage-users`, `manage-settings`, and `manage-roles` into `config/permission-sot.php` and the app-scoped naming contract during Phase 6, and update the already-implemented completed modules to use the new resolver-backed names instead of leaving a split permission model in place.
- **PAT-012**: Reuse the existing multi-role user-management implementation in `app/Http/Controllers/Tyanc/UserController.php`, `app/Actions/Tyanc/Users/StoreUser.php`, `app/Actions/Tyanc/Users/UpdateUser.php`, `app/Data/Tyanc/Users/UserFormData.php`, `resources/js/components/tyanc/users/UserForm.vue`, and the current Tyanc user tests as the baseline; Phase 6 extends that model rather than replacing it.
- **PAT-013**: Replace the current hardcoded frontend app typing (`'tyanc' | 'demo'`) with registry-driven app typing by extending the existing navigation and middleware contract, not by introducing a second app-switch mechanism.
- **PAT-014**: Place Tyanc-only backend orchestration and DTO classes under `app/Actions/Tyanc/**` and `app/Data/Tyanc/**`, while keeping shared platform actions, middleware, models, and DTOs in shared locations.
- **PAT-015**: Keep Tyanc admin route definitions in `routes/tyanc.php` and keep shared public, auth, account, and app-registration bootstrap routes in `routes/web.php` so future app route files can scale without bloating the root web route file.
- **PAT-016**: Place Tyanc-only frontend types under `resources/js/types/tyanc/**`; keep shared shell, navigation, UI, and auth typings in the existing shared `resources/js/types/**` files.
- **PAT-017**: Back all role, permission, and assignment persistence with the already-installed `spatie/laravel-permission` package; do not introduce a parallel custom permission store or a manual permission CRUD model.
- **PAT-018**: Keep `config/permission-sot.php` as the single source of truth for apps, resources, and actions; use the same structure to drive permission syncing, policy checks, and frontend assignment options.
- **PAT-019**: Use a shared permission-name builder or resolver such as `App\Support\Permissions\PermissionKey` so policies combine app, resource, and action segments instead of hardcoding strings.
- **PAT-020**: Sync permission records through an idempotent seeder and/or command that creates missing Spatie permissions from `config/permission-sot.php`, refreshes the permission cache, and does not silently prune production data without an explicit maintenance path.

## 3. Alternatives

- **ALT-001**: Build a decoupled REST or GraphQL backend before building the admin UI. Rejected because the PRD requires Inertia-driven server authority with API readiness rather than API-first duplication.
- **ALT-002**: Use Filament, Nova, or another opinionated admin framework. Rejected because the PRD requires shadcn-vue, Inertia, custom route boundaries, app-aware governance, and full control over UX primitives.
- **ALT-003**: Keep permissions global and unscoped. Rejected because Tyanc must govern multiple apps, and unscoped permission names will collide across future app modules.
- **ALT-004**: Let each future app manage its own roles and permissions separately. Rejected because Tyanc is the central governance plane and must own cross-app RBAC.
- **ALT-005**: Duplicate navigation definitions in the frontend instead of resolving them from `config/sidebar-menu.php` and the app registry. Rejected because navigation must stay aligned with backend permissions and app registration.
- **ALT-006**: Introduce Vue I18n or another frontend translation library. Rejected because the PRD requires Laravel-native JSON translations passed through Inertia as the single translation source.
- **ALT-007**: Allow manual permission CRUD in the database or Tyanc UI. Rejected because Tyanc now uses `config/permission-sot.php` plus Spatie Laravel Permission as the single permission source and sync path.

## 7. Risks & Assumptions

- **RISK-001**: The current app-aware sidebar state can drift if route names, `config/sidebar-menu.php`, app-registry records, middleware sharing, and generated route helpers are changed independently.
- **RISK-002**: Strict schema-to-form parity can create large forms; without careful dialog and sheet composition the UX may become crowded or brittle.
- **RISK-003**: Queue, broadcasting, and PDF infrastructure may require local environment setup beyond the current starter baseline.
- **RISK-004**: Expanding authorization from basic roles and permissions into app registry, a config-driven permission catalog, access matrix, and approval workflows increases the chance of policy drift unless tests stay tight and the sync path remains deterministic.
- **RISK-005**: Contextual translation loading can miss page-specific strings if `ResolveTranslations` and frontend usage drift out of sync.
- **RISK-006**: Locale switching can produce inconsistent shared-route behavior if session locale, authenticated preference, and `current_app` persistence are not resolved in a deterministic order.
- **RISK-007**: App-scoped permission names can become inconsistent if future apps are added without updating `config/permission-sot.php` before running the permission sync workflow.
- **ASSUMPTION-001**: The current multi-app implementation in `routes/web.php`, `routes/tyanc.php`, `config/sidebar-menu.php`, `ResolveSidebarNavigation`, and `useAppNavigation()` is the authoritative baseline for all future work.
- **ASSUMPTION-002**: `/tyanc/dashboard` and `/demo/dashboard` remain the permanent browser entry points unless `config/tyanc.php` changes them explicitly.
- **ASSUMPTION-003**: The `current_app` cookie remains the persistence mechanism for shared-route sidebar context.
- **ASSUMPTION-004**: shadcn-vue remains the only allowed UI primitive source for the implementation.
- **ASSUMPTION-005**: Only `en` and `id` are supported locales unless the PRD is updated again.
- **ASSUMPTION-006**: Phases 1 through 5 are locked baseline work in this revision, and all newly planned governance work begins at Phase 6.
- **ASSUMPTION-007**: Tyanc remains the only governance plane for apps, roles, permissions, and access management even after future apps such as `erp` and `tasks` are introduced.
- **ASSUMPTION-008**: `config/permission-sot.php` remains the canonical catalog for every non-legacy permission exposed by Tyanc and future apps.
