# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan converts `.docs/prds/jeleme-hrm.md` into an executable delivery sequence for building Jeleme as the HRM app that runs under `/jeleme/*`, registers its permissions in Tyanc, and delegates approval review workflows to Cumpu while retaining Jeleme ownership of every HR domain mutation. The plan follows the current Tyanc architecture contract: app registration and page access stay in Tyanc, approval review stays in Cumpu, and Jeleme owns its routes, controllers, actions, DTOs, pages, tests, and database tables. Delivery priority is intentionally front-loaded toward the reusable people foundation: company and org master data, employee records, people directory lookups, work calendars, attendance, leave, and onboarding come first because they are the most likely Jeleme capabilities to be consumed by future apps.

## 1. Requirements & Constraints

- **REQ-001**: Register Jeleme as a first-party app with app key `jeleme`, route prefix `jeleme`, and permission namespace `jeleme` by updating `config/sidebar-menu.php`, `config/permission-sot.php`, and `routes/web.php`.
- **REQ-002**: Place Jeleme web routes in `routes/jeleme.php`, use route names prefixed with `jeleme.`, and keep all Jeleme pages under `resources/js/pages/jeleme/*`.
- **REQ-003**: Keep Jeleme business logic in dedicated action classes under `app/Actions/Jeleme/*`, each exposing a single `handle()` method.
- **REQ-004**: Create typed payload boundaries under `app/Data/Jeleme/*` using Spatie Laravel Data for every Jeleme Inertia page and JSON response.
- **REQ-005**: Use `app/Http/Controllers/Jeleme/*` and `app/Http/Requests/Jeleme/*` for Jeleme HTTP delivery and validation; keep controllers thin and move mutations into actions.
- **REQ-006**: Extend `config/permission-sot.php` so Jeleme has a complete permission surface, including new action verbs required by the PRD such as `submit`, `publish`, `send`, `finalize`, `issue`, and `change`, and make sure `manage_implies` plus `policy_abilities` are explicitly updated for each new verb so policy checks and manage access stay aligned.
- **REQ-007**: Reuse Tyanc app-registry sync, permission sync, and route-helper generation by running `php artisan tyanc:apps-sync --no-interaction`, `php artisan tyanc:permissions-sync --no-interaction`, and `php artisan wayfinder:generate --no-interaction` whenever Jeleme routes, menu entries, or permissions change.
- **REQ-008**: Implement approval-controlled Jeleme mutations through the shared mode-aware approval façade in `App\Actions\Tyanc\Approvals\ExecuteApprovalControlledAction` or its finalized equivalent, keep `SubmitGovernedAction` as the internal `grant`-mode engine behind that façade, and expose state on Jeleme pages through `App\Actions\Tyanc\Approvals\ResolveApprovalContext::handle()`.
- **REQ-009**: Use draft-first domain records plus governed state-transition actions for approval-heavy workflows such as requisition submission, posting publication, offer issuance, payroll finalization, disciplinary issuance, and employee-status changes; do not introduce a second approval replay engine.
- **REQ-010**: Reuse existing Cumpu UI patterns by integrating `resources/js/components/cumpu/approvals/ApprovalRequestBanner.vue`, `ApprovalHistoryPanel.vue`, and `ApprovalReasonDialog.vue` on Jeleme pages that expose governed actions.
- **REQ-011**: Prefix every Jeleme database table with `jlm_`; include `company_id` on core business tables to support multi-company scoping.
- **REQ-012**: Enable soft deletes on employee-related Jeleme tables and use immutable or append-only behavior where the PRD requires auditability, especially payroll runs and career history.
- **REQ-013**: Encrypt sensitive fields such as bank-account values, tax IDs, salary amounts when persisted individually, BPJS identifiers, and other personal finance identifiers through Laravel encrypted casts.
- **REQ-014**: Keep employee master data as the single source of truth reused by payroll, attendance, leave, performance, training, benefits, and offboarding.
- **REQ-015**: Support the recruitment-to-employee conversion path so completing recruitment step 13 creates the employee profile and onboarding records without duplicate manual entry.
- **REQ-016**: Build all Jeleme frontend forms and actions with existing shared shadcn-vue components; do not use plain HTML inputs when shared components already exist.
- **REQ-017**: Use Wayfinder-generated frontend route helpers under `@/routes/jeleme/*` or controller-action helpers under `@/actions/*`; do not hardcode Jeleme URLs in Vue components.
- **REQ-018**: Reuse Tyanc shared file infrastructure for employee documents, payslips, offer letters, warning letters, certificates, and experience letters.
- **REQ-019**: Reuse Tyanc shared notification infrastructure for in-app and email notifications triggered by Jeleme events.
- **REQ-020**: Keep Indonesian (`id`) as the primary Jeleme locale and English (`en`) as the secondary locale by using Laravel translation files and the existing `__()` frontend helper.
- **REQ-021**: Define a deterministic module-to-sidebar and module-to-permission map for dashboard, employees, org, recruitment, onboarding, attendance, leave, payroll, performance, training, benefits, disciplinary, offboarding, reports, and settings so `config/sidebar-menu.php`, `config/permission-sot.php`, and `app_pages` stay synchronized.
- **REQ-022**: Treat Phase 1 through Phase 3 as the Jeleme must-have foundation release: app bootstrap, org structure, employee records, people-directory lookups, work calendar, attendance, leave, and onboarding must be complete before recruitment, payroll, performance, training, disciplinary, offboarding, reporting, or advanced settings work starts.
- **REQ-023**: Provide reusable Jeleme directory and lookup surfaces early for active employees, reporting lines, departments, positions, grades, and companies so future apps can connect to Jeleme as the source of people and organization truth.
- **REQ-024**: For high-input workflows such as employee records, recruitment documents, offer letters, and sensitive HR forms, approval must not block the first save; Jeleme must save a domain draft or editable record first and govern only submit, activate, publish, send, finalize, change, or delete transitions.
- **REQ-025**: Expose a read-only, versioned Jeleme directory contract with dedicated DTOs and permissions for companies, departments, positions, grades, reporting lines, and active employees so future apps can consume people and org data without coupling to internal Jeleme page controllers.
- **REQ-026**: Every Jeleme write action must run through a shared approval detector that resolves one of `none`, `grant`, or `draft` before the action mutates domain state.
- **REQ-027**: Jeleme must integrate only with approval-capable actions that are declared in a shared approval source of truth such as `config/approval-sot.php` so Cumpu approval-rule UI can show only supported governed actions and their configured approval mode.
- **REQ-028**: Define the initial Jeleme approval capability matrix in `config/approval-sot.php` with explicit Cumpu mode selection: use `draft` for manpower-plan submission, job-requisition submission, job-offer issuance, payroll-run finalization, KPI-plan submission, warning-letter issuance, and termination processing; use `grant` for job-posting publication, attendance-correction finalization, overtime-request finalization, leave-decision finalization, external-training enrollment finalization, resignation finalization, and employee-status changes.
- **REQ-029**: Use canonical single-resource permission keys for governed Jeleme actions such as `jeleme.manpower_plans.submit`, `jeleme.job_requisitions.submit`, `jeleme.job_postings.publish`, `jeleme.offer_letters.send`, `jeleme.payroll_runs.finalize`, `jeleme.kpi_plans.submit`, `jeleme.warning_letters.issue`, `jeleme.terminations.process`, `jeleme.attendance_corrections.finalize`, `jeleme.overtime_requests.finalize`, `jeleme.leave_requests.finalize`, `jeleme.training_enrollments.finalize`, `jeleme.resignations.finalize`, and `jeleme.employee_status_changes.change` so `config/permission-sot.php` and `config/approval-sot.php` stay inside the `<app>.<resource>.<action>` contract.
- **SEC-001**: Enforce Tyanc RBAC for app access, page access, and action access; every Jeleme route and sidebar item must map to a valid permission in `config/permission-sot.php`.
- **SEC-002**: Restrict salary, payroll, disciplinary, and personal-identity data to explicit Jeleme permissions and log create, update, delete, and approval-related mutations through the existing activity-log pipeline.
- **SEC-003**: Keep approval review operations in Cumpu; Jeleme must never render its own duplicate approval inbox or review engine.
- **SEC-004**: Secure the attendance-device API with a throttled `routes/api.php` route, request-level shared-secret validation, audit logging, and an explicit secret-rotation path.
- **CON-001**: Do not place Jeleme pages under `resources/js/pages/tyanc/*` or Jeleme business actions under `app/Actions/Tyanc/*` unless the code is truly platform governance.
- **CON-002**: Follow the current repository patterns for routes, tests, settings, DataTable usage, activity logging, and approval context instead of inventing a second Jeleme-specific framework.
- **CON-003**: Keep v1.0 out-of-scope items excluded: AI screening, AI review suggestions, direct job-board APIs, bundled biometric device drivers, LMS content hosting, mobile apps, compensation benchmarking, and payroll bank transfer automation.
- **GUD-001**: Use shared DataTable patterns for Jeleme list pages and shared app-shell conventions for empty states, responsive layout, and navigation behavior.
- **GUD-002**: Keep dialogs and sheets limited to single-screen flows; use full pages for complex HR workflows such as payroll runs, recruitment pipeline management, and employee profile editing.
- **PAT-001**: Model long-running or heavy calculations such as payroll runs, PDF generation, and export generation as queue-backed jobs under `app/Jobs/Jeleme/*`.
- **PAT-002**: Use `DB::transaction()` inside Jeleme actions when a workflow mutates multiple models, especially payroll, recruitment conversion, onboarding triggers, offboarding settlement, and disciplinary issuance.
- **PAT-003**: Add Jeleme tests under `tests/Feature/Jeleme/*` and `tests/Unit/Jeleme/*`, run the smallest relevant Pest suites, then run `composer lint` and `composer test:types` after each implementation slice.
- **PAT-004**: Format dirty PHP files with `vendor/bin/pint --dirty --format agent` before finalizing implementation slices.
- **PAT-005**: Standardize governed-action flow so `Store*` or `Create*` actions create drafts, domain mutation actions such as `Publish*`, `Send*`, `Issue*`, `Change*`, or `Finalize*` call the shared mode-aware approval façade after mode detection, `SubmitGovernedAction` remains the internal `grant`-mode engine, and Cumpu remains the only reviewer surface.
- **PAT-006**: If a Jeleme workflow still proves too heavy for retry-after-approval even with a saved draft, introduce a targeted draft-backed approval extension where Jeleme stores the authoritative draft and Cumpu stores only a draft reference plus immutable reviewer snapshot; do not reintroduce generic payload replay.
- **PAT-007**: Resolve approval mode for every Jeleme write action through a shared approval detector such as `App\Actions\Tyanc\Approvals\DetectApprovalMode` or `ResolveApprovalMode`, backed by a config source such as `config/approval-sot.php`, so Jeleme action classes never guess whether an action uses `none`, `grant`, or `draft` mode.
- **PAT-008**: Keep business status and approval status separate. Use domain lifecycle fields such as `draft`, `submitted`, `active`, or `finalized` only where the business record truly needs them, and use approval morph history plus approval context for cross-cutting review state instead of adding approval columns to every model.

### 1.1 Cumpu approval integration and mode matrix

The table below is the baseline Jeleme approval capability map that must be encoded in `config/approval-sot.php`, surfaced through Cumpu's managed rule screen, and reused by Jeleme pages through shared approval context data.

| ID | Jeleme workflow | Canonical governed permission | Default Cumpu reviewers | Mode | Decision rule |
|----|-----------------|-------------------------------|-------------------------|------|---------------|
| APR-001 | Manpower plan submission | `jeleme.manpower_plans.submit` | HR Director | Draft | Use `draft` because MPP is a multi-line planning artifact and reviewers must approve an exact saved revision. |
| APR-002 | Job requisition submission | `jeleme.job_requisitions.submit` | HR Manager, optional Finance Head | Draft | Use `draft` because requisition details are high-input and must not be re-entered after approval. |
| APR-003 | Job posting publication | `jeleme.job_postings.publish` | HR Manager | Grant | Use `grant` because publishing is a retry-safe state transition on an existing saved posting draft. |
| APR-004 | Job offer issuance | `jeleme.offer_letters.send` | HR Manager | Draft | Use `draft` because offer terms and generated documents must match the approved saved revision. |
| APR-005 | Attendance correction finalization | `jeleme.attendance_corrections.finalize` | Direct Manager or HR | Grant | Use `grant` because the governed step applies a small retry-safe transition on a saved request. |
| APR-006 | Overtime request finalization | `jeleme.overtime_requests.finalize` | Direct Manager | Grant | Use `grant` because approval finalizes a saved request and does not require a new heavy draft lifecycle. |
| APR-007 | Leave decision finalization | `jeleme.leave_requests.finalize` | Direct Manager, optional HR second step | Grant | Use `grant` because the leave request already exists and Cumpu governs only the final decision transition. |
| APR-008 | Payroll run finalization | `jeleme.payroll_runs.finalize` | HR Manager | Draft | Use `draft` because payroll reviewers must approve the exact reviewed run revision without recalculation drift. |
| APR-009 | KPI plan submission | `jeleme.kpi_plans.submit` | Direct Manager | Draft | Use `draft` because KPI plans are multi-goal artifacts whose reviewed revision must be preserved. |
| APR-010 | External training enrollment finalization | `jeleme.training_enrollments.finalize` | Direct Manager, HR | Grant | Use `grant` because approval finalizes a saved enrollment request and remains a retry-safe transition. |
| APR-011 | Warning letter issuance | `jeleme.warning_letters.issue` | HR Manager, Legal | Draft | Use `draft` because disciplinary letters are high-risk documents and must be version-locked for review. |
| APR-012 | Resignation finalization | `jeleme.resignations.finalize` | HR Manager | Grant | Use `grant` because the governed step is the final decision on an existing resignation request, not a new heavy draft save. |
| APR-013 | Termination processing | `jeleme.terminations.process` | HR Manager, Legal, Director | Draft | Use `draft` because termination is legally sensitive and tied to exact grounds, dates, and settlement context. |
| APR-014 | Employee status change | `jeleme.employee_status_changes.change` | HR Manager | Grant | Use `grant` because the mutation is a small retry-safe state change on an existing employee record. |

- **MODE-001**: For every `grant` row above, Jeleme must persist the request or proposal record first and let Cumpu govern only the final retry-safe transition.
- **MODE-002**: For every `draft` row above, Jeleme must persist an explicit draft subject first, submit the saved revision to Cumpu, and commit only the approved revision through Jeleme-owned domain actions.

## 3. Alternatives

- **ALT-001**: Build Jeleme as a big-bang HR suite in one release branch. Rejected because the PRD has hard master-data dependencies and payroll or recruitment defects would block the entire app.
- **ALT-002**: Build a Jeleme-local approval inbox and approval tables. Rejected because Tyanc already defines Cumpu as the only approval workspace and `SubmitGovernedAction` as the governed-action contract.
- **ALT-003**: Store employee profile sections in one wide JSON column. Rejected because payroll, leave, attendance, recruitment conversion, and reporting need normalized relations and queryable fields.
- **ALT-004**: Treat approval-heavy submissions as payload replay workflows. Rejected because the current Tyanc approval model explicitly favors governed re-run or governed transition actions over a second replay engine.
- **ALT-005**: Use the same retry-after-approval pattern for every Jeleme create flow, including large employee and recruitment forms. Rejected because high-input HR workflows need first-save drafts and governed transitions, not forced full-form re-entry.

## 7. Risks & Assumptions

- **RISK-001**: Payroll compliance for PPh 21, BPJS, and severance can invalidate downstream calculations if stakeholder formulas are not signed off early.
- **RISK-002**: Recruitment scope is large enough to stall delivery if the 13-step pipeline is implemented before master data and employee conversion contracts are stable.
- **RISK-003**: Report pages can miss the PRD performance targets unless list and aggregation queries are indexed and load-tested during each module phase.
- **RISK-004**: Approval UX can become confusing if Jeleme pages do not clearly explain governed actions, pending requests, one-time grants, and draft-commit states.
- **RISK-005**: Jeleme `draft` mode work can stall if the shared Cumpu capability-mode foundation is delayed or diverges from the plan Jeleme expects.
- **ASSUMPTION-001**: Jeleme domain models can live under `app/Models/Jeleme/*` without conflicting with existing platform models.
- **ASSUMPTION-002**: Existing Tyanc and Cumpu infrastructure is stable enough that Jeleme only needs integration work, not platform rewrites.
- **ASSUMPTION-003**: `company_id` scoping is sufficient for Jeleme v1.0 multi-entity requirements.
- **ASSUMPTION-004**: Out-of-scope items listed in the PRD remain deferred and will not be folded into Jeleme v1.0 during implementation.
